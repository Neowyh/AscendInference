# 评测系统分支审查报告

## 分支概况

- 审查分支：`codex/ascend-yolo-system`
- 审查日期：`2026-03-31`
- 分支定位：在推理工具基础上扩展评测、注册表、报告与 smoke 能力的主功能分支
- 相对 `master` 的主要差异：
  - 约 47 个文件级改动
  - 新增 `evaluations/`、`registry/`、`reporting/`
  - 扩展 CLI、评测配置、smoke 脚本和测试覆盖

## 审查范围与证据

- 检查 `main.py` 的命令扩展与 CLI 接口
- 审查 `evaluations/`、`registry/`、`reporting/` 的职责边界
- 对比 `config/config.py`、`config/validator.py` 与评测配置 JSON
- 检查 `scripts/run_smoke_eval.py` 与相关 smoke 配置
- 结合现有测试结果，评估新增能力的回归风险与维护成本

## 高优先级问题

### 1. 命令入口扩展快于文档同步，增加了使用和审查成本

观察：

- `main.py` 已经引入 `model-bench`、`strategy-bench`、`extreme-bench`
- 但命令帮助、README、实现说明和协作说明曾经没有同步更新

影响：

- 用户难以判断命令参数的稳定性
- 审查时很难确认文档与运行时是否表达同一套行为

建议动作：

- 将 CLI 帮助、README、`docs/00-文档导航.md` 与实现说明作为同一批次更新

### 2. 新增评测、注册表与报告能力需要更清晰的验收矩阵

观察：

- `evaluations/` 负责路线、输入分层和任务编排
- `registry/` 提供设备、模型与场景契约
- `reporting/` 负责标准报告模型、渲染和归档
- 三者协作链路变长，但此前缺少按子域拆分的验收矩阵

影响：

- 测试即使覆盖了部分 happy path，也难以证明不同子域之间的边界是否稳定
- 新增问题容易在模块交界处出现

建议动作：

- 分别建立：
  - 评测任务编排验收
  - 注册表契约验收
  - 报告渲染与归档验收
- 再补充贯穿三者的端到端 smoke 验收

### 3. 分辨率与路线词汇存在多源定义，容易造成配置和运行时语义漂移

涉及位置：

- `config/config.py`
- `config/validator.py`
- `evaluations/tiers.py`
- 评测配置 JSON
- CLI 文案

影响：

- 同一分辨率概念可能在配置、提示信息和运行时映射中出现不同表达
- 用户输入和内部执行语义之间容易产生偏差

建议动作：

- 建立统一词汇表，并在配置、校验、CLI 与文档中同步收敛

## 中优先级问题

### 1. `registry/` 与旧有 benchmark/场景定义之间需要更明确的职责边界

观察：

- 新分支引入注册表风格的数据契约
- 旧有 benchmark 场景层仍然承担部分装配职责

影响：

- 如果边界不清晰，未来容易出现重复建模和双重真相源

建议动作：

- 在实现说明中明确：
  - 哪些信息由注册表定义
  - 哪些信息由评测任务编排层消费
  - 哪些信息仅在报告层使用

### 2. smoke 脚本提供了很好的硬件验证入口，但需要区分“预览”与“真实执行”

观察：

- `scripts/run_smoke_eval.py` 同时支持命令预览和带 `--run` 的真实执行

影响：

- 在无 Ascend 硬件环境下，如果没有清楚区分，会让使用者误以为预览即等于执行成功

建议动作：

- 在文档中明确：
  - 默认模式只生成命令
  - `--run` 才代表真实执行
  - 哪些结果需要人工检查报告产物

## 低优先级问题

### 1. 分支规模较大，单次审查负担偏重

约 47 个文件级改动对单次合并审查来说已经不算小。

建议动作：

- 合并时尽量按主题拆分审查，至少把文档、配置、核心逻辑和 smoke 工具分别核对

## 测试与验证结论

- 分支方向正确，新增能力与项目目标高度一致
- 但在合并前，需要先处理：
  - 基线是否稳定
  - 命令与文档是否同步
  - 子域验收矩阵是否完整
- 当前更适合在“加固后合并”，而不是直接落主线

## 合并风险

- 风险等级：中高
- 风险主要来自：
  - 改动面广
  - 子域较多
  - 配置、运行时和文档存在同步成本

## 建议动作

1. 在稳定 `master` 基线后再推进合并
2. 统一 CLI、配置词汇和文档表达
3. 为评测、注册表、报告和 smoke 建立分层验收矩阵
4. 合并审查时按主题拆开核对，而不是一次性只看总文件数
