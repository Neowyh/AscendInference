## 高优先级问题

### 1. 基线测试未通过，`master` 不能直接作为可靠的合并目标

`2026-03-31` 执行 `pytest tests -q` 的结果为：

- `201 passed / 13 failed / 4 errors`
- 环境噪音：
  - `C:\Users\18222\AppData\Local\Temp\pytest-of-18222` 下的 `PermissionError`
  - 工作区内 `.pytest_cache` 写入相关警告
- 呈现为真实行为偏差的问题：
  - `src/api.py` 在导入可用性检查之前先做路径校验
  - `src/inference/base.py` 在 ACL 可用性检查之前先做路径校验
  - `utils/logger.py` 的日志器复用与输出捕获行为和测试预期不一致

影响：

- 在基线为红的情况下继续合并新功能，会让缺陷归因变得模糊。
- 当前失败已经同时混入环境噪音和真实行为问题，不适合作为后续验证基准。

建议动作：

- 将“基线测试稳定化”设为合并前的零号门槛。
- 把问题拆成两类处理：
  - 环境相关测试夹具问题
  - 运行时与测试语义不一致问题

观察到的漂移包括：

- 根目录 `AGENTS.md` 仍以旧版架构为中心，没有及时反映评测、注册表与报告模块

影响：

- 若代理说明不进入版本控制，会持续削弱后续评审质量。

建议动作：

- 在主线准备合并前，先把更新后的 `AGENTS.md` 纳入版本控制。

## 中优先级问题

### 1. 测试预期与运行时顺序正在逐步脱节

典型现象：

- `InferenceAPI.inference_image()` 和 `inference_batch()` 会先校验文件，再判断 `HAS_INFERENCE`
- `Inference.preprocess()` 会先校验输入路径，再判断 `HAS_ACL`

影响：

- 以导入失败或 ACL 不可用为目标的测试，可能提前被路径校验短路
- 错误优先级变得依赖环境状态，难以维护与解释

建议动作：

- 明确以下优先级语义：
  - 模块是否可用
  - ACL 是否可用
  - 路径是否存在
- 然后同步调整代码与测试

### 2. 日志初始化能工作，但对测试和维护不够稳定

在 `utils/logger.py` 中观察到：

- 命名 logger 若已有 handler，会直接复用现有实例
- `propagate` 固定为 `False`
- 后续调用即便改变级别、文件目标或格式，也不一定重新应用配置

影响：

- `capsys` 与 `caplog` 相关测试容易出现顺序依赖
- 重复初始化调用无法可靠覆盖旧配置
- 后续分支很容易继承不可见的日志状态

建议动作：

- 重构 logger 初始化流程，使重复配置可预测、可测试、可覆盖

## 低优先级问题

### 1. Windows shell 中存在编码显示噪音

仓库内容主要使用 UTF-8 中文，但在部分 Windows shell 会出现乱码显示。

影响：

- 对运行时风险较低
- 对 Windows 审查与维护体验有一定干扰

建议动作：

- 保持 UTF-8 编码不变
- 在协作文档中补充终端和编辑器的编码约定

## 测试与验证结论

- `master` 目前不能视为干净的稳定基线
- 在作为可靠合并目标之前，必须先完成一轮稳定化处理
- 不是所有失败都属于代码回归，必须区分：
  - 环境噪音
  - 基线缺陷
  - 功能分支缺陷

## 合并风险

- 风险等级：高
- 风险原因并不是 `master` 改动过大，而是它缺少可信的绿色验证状态
- 如果直接在这种基线上合并 `codex/ascend-yolo-system`，后续失败归因会非常模糊

## 建议动作

1. 先稳定基线测试，再合并功能分支
2. 将更新后的 `AGENTS.md` 正式纳入主线
3. 明确导入检查、ACL 检查与路径校验的错误优先级
4. 重构日志初始化，使其可重复、可测试、可维护
