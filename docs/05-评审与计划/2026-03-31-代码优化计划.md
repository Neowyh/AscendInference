# 代码优化计划

## 目标

本计划用于指导三分支合并后的代码整理与质量提升工作，重点解决基线不稳定、错误优先级不清晰、日志初始化脆弱、评测子域边界不够清晰以及文档漂移等问题。

本轮计划只定义后续任务，不承诺引入新的运行时接口。

## 优化波次

### 波次 A：修复测试基线与环境噪音

目标：

- 重新建立可信的 `pytest tests -q` 基线
- 区分环境噪音与真实行为缺陷

重点任务：

- 识别并隔离 Windows 临时目录权限导致的 `PermissionError`
- 清理 `.pytest_cache` 和临时目录相关的测试干扰
- 为基线失败建立分类清单：环境问题、旧缺陷、新缺陷

验收标准：

- 测试结果能够稳定复现
- 失败项被明确归类，而不是混杂在一起

### 波次 B：修复推理 API 与校验顺序不一致问题

目标：

- 明确导入检查、ACL 检查与路径校验的错误优先级

重点任务：

- 审查 `src/api.py`
- 审查 `src/inference/base.py`
- 对齐测试夹具和运行时代码对错误优先级的理解

验收标准：

- 公共接口在导入不可用、ACL 不可用和路径非法时，能稳定给出预期异常
- 测试用例不再因环境差异出现语义漂移

### 波次 C：修复日志捕获与测试断言不稳定问题

目标：

- 让日志初始化和重复配置行为可预测、可测试

重点任务：

- 审查 `utils/logger.py` 中 logger 复用逻辑
- 明确 handler 清理、`propagate`、输出目标与格式器覆盖策略
- 修复 `caplog`、`capsys` 相关测试的顺序依赖

验收标准：

- 重复调用日志初始化接口时，行为一致且可覆盖
- 日志相关测试在独立与全量执行场景下都稳定

### 波次 D：梳理评测、注册表、报告模块之间的边界

目标：

- 降低 `evaluations/`、`registry/`、`reporting/` 之间的重复建模与职责耦合

重点任务：

- 明确哪些数据由注册表定义
- 明确哪些任务由评测编排层负责
- 明确哪些字段仅用于报告渲染和归档
- 建立子域级验收矩阵，覆盖：
  - 评测任务编排
  - 注册表契约
  - 报告渲染
  - 归档布局

验收标准：

- 实现说明能够明确解释三者关系
- 子域测试边界清晰，不重复验证同一职责

### 波次 E：补齐 README、文档与命令帮助文本

目标：

- 让用户入口、实现说明和协作说明保持同步

重点任务：

- 更新 `README.md`
- 更新 `docs/00-文档导航.md`
- 更新 `AGENTS.md`
- 对齐 `main.py --help` 与各子命令帮助文本

验收标准：

- 命令、配置、评测路线和报告行为在文档中都有准确入口
- 新增能力不会只存在于代码中而缺少说明

## 首批优化对象

- `src/api.py`、`src/inference/base.py`
  - 修复参数校验与 ACL 检查顺序导致的语义漂移
- `utils/logger.py`
  - 修复 logger 复用、`propagate` 与输出捕获行为导致的测试脆弱
- `AGENTS.md`、`README.md`、`docs/`
  - 消除文档与实际结构不一致的问题
- `codex/ascend-yolo-system` 引入的评测能力
  - 按子域建立更清晰的验收边界

## 实施顺序建议

1. 先处理波次 A，建立可信基线
2. 再处理波次 B 和 C，修复最直接影响测试稳定性的运行时问题
3. 接着推进波次 D，清理新增子域边界
4. 最后完成波次 E，统一对外说明

## 风险提示

- 如果跳过波次 A，后续所有优化都难以判断真实收益
- 如果不先收敛错误优先级和日志初始化语义，测试仍会持续出现“偶发失败”
- 如果不建立子域验收矩阵，评测系统后续扩展会越来越难维护

## 结论

本计划建议先稳定，再收敛，再扩展。只有把基线、错误语义、日志行为和文档同步做好，评测系统和推理优化能力才能长期可靠演进。
