# AscendInference 昇腾端侧 YOLO 选型与优化系统设计

- 日期: 2026-03-28
- 状态: Draft
- 适用阶段: 第一阶段设计基线

## 1. 设计目标

本设计将 `AscendInference` 从通用昇腾推理工具，演进为面向昇腾端侧设备的 YOLO 标准化评测与优化系统工程。

系统第一阶段同时服务两类核心用户：

- 算法工程师：需要在多模型、多输入规模、多场景下完成模型选型。
- 部署优化工程师：需要把端侧优化策略拆分、组合、评测，并为部署策略选择提供依据。

第一阶段的核心交付物是标准报告，而不是单次推理结果。系统应持续产出可复现、可比较、可归档的性能与资源分析结果。

## 2. 第一阶段范围

第一阶段聚焦评测与优化闭环，不扩展到训练和平台化能力。

### 2.1 范围内

- YOLO `.om` 模型资产管理
- 标准评测主线
- 高分辨率遥感主线
- 端侧优化策略单元化与组合评测
- 性能与资源指标统一采集
- 标准报告生成与结果归档

### 2.2 暂不纳入

- 检测效果指标正式验收，如 `mAP`、`precision`、`recall`
- 训练、蒸馏、自动量化、自动模型转换
- Web 平台与多用户协作系统
- 数据库化资产管理

## 3. 系统定位

系统目标不是单次命令执行，而是围绕一次评测任务形成完整闭环：

1. 注册模型、设备、场景与任务资产
2. 按标准矩阵执行评测
3. 比较模型与策略组合
4. 输出标准化报告
5. 沉淀可复用的归档结果

系统应回答两类问题：

1. 在某个端侧设备和输入场景下，哪个 YOLO 模型更合适。
2. 在某个端侧设备和输入场景下，哪套推理优化策略更平衡。

## 4. 总体分层架构

系统按 5 层组织。

### 4.1 资产层

负责管理评测输入资产。

- 模型清单
- 设备清单
- 场景清单
- 输入分档定义
- 任务模板

第一阶段采用 `YAML/JSON + 目录约定` 即可，不引入数据库。

### 4.2 执行层

负责实际推理、策略执行、路线对照实验和结果记录。

执行层不再只是“推理模块”，而是一个服务模型选型和策略实验的执行引擎。

### 4.3 指标层

统一采集和归一化指标，包括：

- 单次推理时延
- 端到端总时延
- 吞吐
- CPU 占用
- 内存占用
- NPU 占用
- 峰值资源占用
- 连续运行稳定性

### 4.4 报告层

负责将任务结果转为标准报告。

第一阶段至少输出：

- 面向归档的 `JSON`
- 面向阅读的 `Markdown` 或 `HTML`

### 4.5 编排层

负责任务定义与执行组织，包括：

- 选择模型集合
- 选择场景
- 选择输入分档或图幅档位
- 选择执行路线
- 选择策略组合
- 触发执行并归档

## 5. 执行层完整设计

执行层采用“两条主线、两类路线、统一结果出口”的设计。

### 5.1 标准评测主线

面向常规端侧场景，目标是在标准化条件下完成不同模型的公平对比。

第一阶段显式考虑输入图像分档，固定为：

- `720p`
- `1080p`
- `4K`

这条主线评测的重点不是固定输入 `.om` 模型在不同原图尺寸下的裸推理差异，而是评估真实端到端链路在不同输入档位下的成本变化。

其核心评测矩阵定义为：

`模型 x 输入分档 x 策略组合`

报告中必须显式区分：

- 模型执行指标
- 系统端到端指标

### 5.2 高分辨率遥感主线

面向 `6K` 等遥感大图场景，目标是在大图条件下比较不同工程路线在性能和资源上的权衡。

这条主线保留两条并行路线：

- `tiled_route`
  - 使用常规固定输入尺寸模型
  - 对大图做滑窗切片
  - 逐 tile 推理
  - 全图回拼

- `large_input_route`
  - 使用固定大输入尺寸模型，例如 `6K x 6K`
  - 整图直接推理
  - 不进行 resize

两条路线分别代表：

- 用执行策略解决高分辨率问题
- 用模型规格提升解决高分辨率问题

高分辨率遥感主线的核心评测矩阵定义为：

`路线类型 x 模型 x 图幅档位 x 策略组合`

第一阶段图幅先聚焦：

- `6K`

后续可扩展到 `8K` 或更大图幅。

## 6. 执行层内部模块

执行层建议拆成 7 个模块。

### 6.1 `Execution Core`

负责纯推理生命周期：

- ACL 上下文管理
- 模型加载与释放
- 输入输出 buffer 管理
- 单次推理调用

该模块不感知模型对比、路线实验和报告逻辑。

### 6.2 `Scenario Adapter`

把不同输入场景统一成标准任务单元：

- 单图输入
- 批量图输入
- `720p / 1080p / 4K` 输入分档
- `6K` 遥感大图输入

### 6.3 `Model Evaluation Orchestrator`

服务标准评测主线，负责：

- 多模型评测
- 多输入分档执行
- 基线结果采集
- 统一预热和迭代控制

### 6.4 `Route Experiment Orchestrator`

服务高分辨率遥感主线，负责：

- `tiled_route` 与 `large_input_route` 对照实验
- 路线内策略组合实验
- 全图级指标汇总

### 6.5 `Strategy Unit Interface`

定义所有优化策略的统一接口，使其具备：

- 独立启停能力
- 独立评测能力
- 可组合能力
- 可解释收益能力

### 6.6 `Strategy Composition Engine`

负责：

- 策略依赖解析
- 策略冲突检测
- 策略组合装配

### 6.7 `Execution Recorder`

统一记录执行结果，并向指标层和报告层输出标准结构。

结果至少包含：

- 模型信息
- 路线信息
- 场景信息
- 输入档位或图幅档位
- 策略组合
- 原始执行指标
- 环境快照
- 异常与告警

## 7. 策略系统设计

策略拆分原则为：只要一个优化手段能独立开关、独立评测、独立解释收益，就应成为独立策略单元。

### 7.1 第一阶段策略分组

#### 调度类

- `single_thread`
- `multithread`
- `pipeline`

#### 批处理类

- `fixed_batch`
- 后续扩展 `dynamic_batch`

#### 输入处理类

- `parallel_preprocess`
- `high_res_tiling`

#### 资源复用类

- `memory_pool`
- 后续扩展 `executor_pool`
- `buffer_reuse`

#### 回拼与融合类

- tile 回拼
- 跨 tile 融合

### 7.2 第一阶段优先落地策略

- `multithread`
- `fixed_batch`
- `pipeline`
- `high_res_tiling`
- `memory_pool`
- `parallel_preprocess`

## 8. 数据流与任务流程

系统任务流程固定为 6 步：

1. 注册资产  
   模型、设备、场景、输入分档、图幅档位进入资产层。

2. 创建评测任务  
   描述模型集合、场景、路线、策略组合和输出位置。

3. 执行基线评测  
   先跑无复杂策略的标准结果。

4. 执行策略或路线实验  
   跑不同策略组合，或在遥感场景下跑不同路线。

5. 汇总与归一化  
   所有结果转成统一结构。

6. 生成与归档报告  
   输出标准报告，并保留配置快照、原始指标和日志。

## 9. 指标体系

### 9.1 标准评测主线指标

- 单次推理时延
- 端到端总时延
- 吞吐
- CPU 占用
- 内存占用
- NPU 占用
- 峰值资源占用
- 连续运行稳定性

### 9.2 高分辨率遥感主线指标

除常规指标外，还需记录：

- 全图总时延
- 单 tile 平均时延
- tile 数量
- 预处理阶段总耗时
- 推理阶段总耗时
- 回拼与融合耗时

第一阶段暂不将检测效果指标纳入正式验收，但执行层和报告结构需为后续 `mAP`、`precision`、`recall` 扩展留出字段。

## 10. 报告设计

第一阶段报告至少包含以下部分：

- 任务摘要
- 测试环境
- 模型清单
- 场景说明
- 输入档位说明
- 路线说明
- 模型对比表
- 策略对比表
- 路线对比表
- 关键指标汇总
- 结论建议
- 附录：日志与配置快照索引

其中：

- 标准评测主线重点展示 `720p / 1080p / 4K`
- 高分辨率遥感主线重点展示 `tiled_route` 与 `large_input_route`

## 11. 错误处理与测试策略

### 11.1 错误处理分类

- 资产错误
- 执行错误
- 指标错误
- 报告错误

### 11.2 测试分层

#### 单元测试

- 配置解析
- 策略单元
- 指标聚合
- 报告生成

#### 集成测试

- 从任务配置到报告产出的完整链路

#### 无硬件 mock 测试

- 验证任务编排
- 验证路线对照逻辑
- 验证报告链路

## 12. 目录演进建议

建议在现有仓库基础上演进出如下职责分布：

- `src/`
  - 保留推理和策略底座
- `benchmark/`
  - 保留场景执行能力
- `config/`
  - 扩展任务和场景配置
- `reports/` 或 `reporting/`
  - 新增报告模型和渲染器
- `assets/` 或 `registry/`
  - 新增模型、设备、场景定义
- `evaluations/` 或 `workflows/`
  - 新增评测任务编排

## 13. 第一阶段验收重点

第一阶段的核心验收不以检测效果为中心，而以标准报告和执行闭环为中心。

至少应满足：

- 能对多个 YOLO `.om` 模型完成标准评测
- 能在 `720p / 1080p / 4K` 下输出标准化对比结果
- 能在 `6K` 遥感场景下完成 `tiled_route` 与 `large_input_route` 的对照实验
- 能对核心策略单元进行独立和组合评测
- 能统一输出性能与资源分析报告
- 能归档配置、日志和原始结果

## 14. 后续扩展方向

第一阶段之后，可按以下方向演进：

- 将检测效果指标接入正式评测
- 扩展遥感图幅档位
- 扩展更多端侧优化策略
- 引入任务数据库和历史比较能力
- 增加 Web 可视化入口
